# Omnidapter Server: Functional Specification

The **Omnidapter Server** is a RESTful API service that wraps the `omnidapter-core` library. It provides a persistent, multi-tenant environment for managing third-party connections and interacting with their services over HTTP.

**License**: MIT (Open Source)

---

## 1. Overview & Architecture

Omnidapter Server acts as a middle layer between your application and the Omnidapter Core engine. It handles database persistence, request authentication, and provides a standardized API for connection management.

### 1.1 Multi-Tenancy (API Level)
The server is designed to support multiple organizations (`org_id`).
*   **Isolation**: Resources like Connections and Provider Configurations are scoped to an organization.
*   **Global Access**: Admin-level access is provided via API Keys for server-to-server communication.

### 1.2 Core Integration
The server uses a database-backed `CredentialStore` and `OAuthStateStore`, ensuring that user authorizations survive server restarts and scale across multiple instances.

## 2. Connection Management

### 2.1 The Connection Lifecycle
A `Connection` in the server represents a stored authorization between an end-user and a provider.
*   **Statuses**:
    *   `PENDING`: Interaction started but not yet authorized.
    *   `ACTIVE`: Healthy and ready for service calls.
    *   `NEEDS_REAUTH`: Token refresh failed or was revoked; user must re-interact.
    *   **Revoked**: Explicitly disconnected by the user or provider.
*   **Health Tracking**: Automatically updates `last_used_at` and performs health checks during service proxying.

### 2.2 Link Tokens
To securely initiate a connection flow from a frontend (like the Connect UI) without exposing sensitive configuration:
1.  The backend requests a `Link Token` with specific constraints (allowed providers, metadata, redirect URIs).
2.  The frontend uses this token to initialize the Omnidapter flow.
3.  The server verifies the token via bcrypt hash lookup, checks expiration, and validates constraints before allowing the OAuth flow to complete.

---

## 3. Service Proxying

Omnidapter Server exposes Core services (like Calendar) via a proxy mechanism.

### 3.1 Calendar API
Standardized endpoints for performing calendar operations:
*   `GET /v1/connections/{id}/calendars`: Lists available calendars.
*   `GET /v1/connections/{id}/calendars/{cal_id}/events`: Returns a normalized list of events.
*   `POST /v1/connections/{id}/calendars/{cal_id}/events`: Unified event creation across all providers.

### 3.2 Benefits of Proxying
*   **Homogenization**: Client applications receive the same JSON schema regardless of whether the underlying provider is Google, Microsoft, or Apple.
*   **Automatic Refreshing**: The server handles OAuth2 token refreshing transparently during the API call.
*   **Error Mapping**: Provider-specific errors are mapped to standard HTTP status codes (401, 403, 429, etc.) with consistent error bodies.

---

## 4. Authentication & Security

### 4.1 API Key Authentication
Access to the server API is secured via API Keys.
*   **Lifecycle**: Keys can be created, listed, and revoked via the API.
*   **Scope**: Keys are typically associated with an organization but can be granted global permissions for administration.

### 4.2 Credentials Encryption
Sensitive tokens (Access/Refresh tokens) are encrypted before being stored in the database, protecting against snapshots or unauthorized database access.

---

## 5. Provider Configuration

Allows organizations to bring their own OAuth credentials (Client ID/Secret) for specific providers.
*   **Overrides**: Organizations can define their own `OAuthConfig` which the server will prefer over global environment variables.
*   **Validation**: The server validates the configuration (e.g., checking if the authorization endpoint is reachable) before saving.

---

## 6. Usage & Monitoring

*   **Usage Recording**: The server tracks API calls and connection activity, providing metadata useful for analytics or billing in a hosted environment.
*   **Audit Logging**: Basic logging of connection status changes and authentication failures.

---

## 7. Database Migrations

### Critical Rule

⚠️ **NEVER manually edit migration files.** Migrations must be autogenerated from SQLAlchemy models. Manual edits cause schema/model mismatches, break audits, and introduce subtle bugs that are hard to debug.

**Correct approach:**
1. Update the SQLAlchemy model in `omnidapter-server/src/omnidapter_server/models/`
2. Run `alembic revision --autogenerate -m "description"` to generate the migration
3. Review and commit the generated migration

Autogenerate compares your models against the current database schema and produces the exact SQL needed. It's the source of truth for what changed and why.
